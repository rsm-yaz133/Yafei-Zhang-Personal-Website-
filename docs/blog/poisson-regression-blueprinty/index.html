Does Blueprinty’s software actually help firms get more patents?

Building a Poisson regression from scratch to pressure-test a software company’s marketing claim. Spoiler: the answer is murkier than the sales deck suggests.

The pitch (and why I’m skeptical)

Blueprinty makes software for developing blueprints to submit patent applications to the US Patent and Trademark Office. Their marketing team wants to claim that firms using their product get more patents approved. That’s a great story to tell prospective customers. But is it actually true?

Here’s my concern. Blueprinty didn’t run an experiment. They didn’t randomly assign their software to half the firms and withhold it from the other half. They also don’t have before-and-after data on the same firms. What they have is a snapshot: 1,500 engineering firms, some of whom happen to be Blueprinty customers and some who don’t. For each firm, we know how many patents they were awarded over five years, where they’re located, how old they are, and whether they use the software.

The problem is obvious once you think about it. Firms that buy specialized patent software are probably not the same kinds of firms that don’t. Maybe they’re already patent-heavy. Maybe they’re in regions with more R&D activity. Maybe they’re just bigger. If that’s the case, then any difference in patent counts between the two groups could have nothing to do with the software.

So I’m going to build a Poisson regression model from scratch, fit it with maximum likelihood, and see what falls out. I’ll control for what I can. But I want to be upfront: controlling for observables is not the same as establishing causation.

Looking at the data

We have 1,500 firms. Let’s start with the most basic comparison: do customers get more patents?

If you stopped here, you might conclude the software works. But we shouldn’t stop here. The two groups might differ in ways that have nothing to do with Blueprinty.

Where are these firms?

This is the part that made me uneasy. Blueprinty’s customer base is overwhelmingly concentrated in the Northeast. Non-customers are spread more evenly. If Northeast firms patent more (because of proximity to tech corridors, more R&D spending, whatever the reason), that regional concentration alone could explain the patent gap. The software might have nothing to do with it.

How old are these firms?

Customers average 26.9 years old, non-customers 26.1.

The age gap is smaller than the regional one, but it’s there. Older firms have had more time to build patent portfolios, and they might also be more likely to invest in specialized tools. Another potential confound.

So the raw comparison is polluted by at least two things: geography and age. We need a model that can hold those constant.

Starting simple: one-parameter Poisson

Why Poisson?

Our outcome is a count of patents. Counts can’t be negative and they have to be integers. A Normal distribution doesn’t make sense here (it’s continuous and allows negative values). The Poisson distribution is the natural starting point for count data.

A Poisson random variable \(Y\) with rate \(\lambda > 0\) has this probability mass function:

\[ f(Y | \lambda) = \frac{e^{-\lambda} \lambda^Y}{Y!} \]

The parameter \(\lambda\) is both the mean and the variance. (That equal-mean-and-variance assumption is actually quite restrictive, but it’s a reasonable place to start.)

Deriving the log-likelihood

If we have \(n\) independent observations \(Y_1, \ldots, Y_n\), the joint likelihood is the product of individual probabilities:

\[ L(\lambda) = \prod_{i=1}^{n} \frac{e^{-\lambda} \lambda^{Y_i}}{Y_i!} \]

Taking logs turns the product into a sum, which is much easier to work with:

\[ \ell(\lambda) = \sum_{i=1}^{n} \left[ -\lambda + Y_i \log(\lambda) - \log(Y_i!) \right] \]

We can pull terms apart:

\[ \ell(\lambda) = -n\lambda + \log(\lambda) \sum_{i=1}^{n} Y_i - \sum_{i=1}^{n} \log(Y_i!) \]

That last term doesn’t depend on \(\lambda\) at all, so the optimizer can ignore it.

Coding it up

My first attempt used math.factorial, which works but gets shaky with large counts:

Code 
def poisson_loglikelihood(lam, Y):
    """Log-likelihood for Y ~ Poisson(lambda), naive version."""
    n = len(Y)
    ll = -n * lam + np.sum(Y) * np.log(lam) - np.sum(np.log(np.array([np.math.factorial(int(y)) for y in Y])))
    return ll

A cleaner approach uses gammaln from scipy, since \(\log(Y!) = \texttt{gammaln}(Y+1)\):

Code 
from scipy.special import gammaln

def poisson_loglikelihood(lam, Y):
    """Log-likelihood for Y ~ Poisson(lambda), stable version."""
    return np.sum(-lam + Y * np.log(lam) - gammaln(Y + 1))

The curve is concave and peaks at the sample mean. You can read the MLE right off the plot.

We can verify this analytically

Take the derivative, set it to zero:

\[ \frac{d\ell}{d\lambda} = -n + \frac{\sum Y_i}{\lambda} = 0 \quad \Longrightarrow \quad \hat{\lambda}_{\text{MLE}} = \frac{\sum Y_i}{n} = \bar{Y} \]

The MLE for a Poisson is just the sample mean. That should feel right, since \(\lambda\) is the mean of the distribution.

Confirming numerically

Code 
result = optimize.minimize_scalar(lambda lam: -poisson_loglikelihood(lam, Y), bounds=(0.1, 20), method='bounded')

print(f"Numerical MLE:  {result.x:.4f}")
print(f"Sample mean:    {Y.mean():.4f}")
Numerical MLE:  3.6847
Sample mean:    3.6847

They match. Good. The code is doing what the math says it should.

Poisson regression: letting each firm have its own rate

The idea

The one-parameter model assumes every firm has the same patent rate \(\lambda\). That’s clearly wrong. A 10-year-old firm in the Southwest is not going to patent at the same rate as a 35-year-old firm in the Northeast.

Poisson regression lets each firm have its own rate:

\[ Y_i \sim \text{Poisson}(\lambda_i) \qquad \text{where} \qquad \lambda_i = \exp(X_i' \beta) \]

Why the exponential? Because \(\lambda_i\) has to be positive, but the linear combination \(X_i'\beta\) can be any real number. The exponential maps the whole real line onto strictly positive values, so \(\lambda_i\) is always valid.

Building the covariate matrix

A few choices worth explaining:

I included age-squared because the relationship between age and patenting probably isn’t linear. Very young firms are still getting started, very old firms may have shifted focus. A quadratic lets the model capture that hump shape.

I dropped the Northeast region indicator. If you include every region plus an intercept, the columns are perfectly collinear (they add up to the intercept). You have to leave one out. Northeast is the reference category, so the other region coefficients measure the difference from Northeast.

And then there’s iscustomer, the variable we actually care about.

Updating the log-likelihood

Code 
def poisson_regression_loglikelihood(beta, Y, X):
    """Log-likelihood for Poisson regression: Y_i ~ Poisson(exp(X_i @ beta))."""
    X_arr = np.array(X)
    xb = X_arr @ beta
    xb = np.clip(xb, -20, 20)  # prevent overflow during optimization
    lambda_i = np.exp(xb)
    ll = np.sum(Y * np.log(lambda_i) - lambda_i - gammaln(Y + 1))
    return ll

I added a clip on the linear predictor. Without it, the optimizer can wander into regions where exp(X @ beta) overflows to infinity, which kills the whole run. Clipping at +/- 20 keeps things stable without affecting the final estimates.

Finding the MLE

Code 
# Start the intercept near log(mean), everything else at zero
beta_init = np.zeros(X.shape[1])
beta_init[0] = np.log(Y.mean())

X_arr = np.array(X)

def neg_ll(b):
    return -poisson_regression_loglikelihood(b, Y, X_arr)

result = optimize.minimize(
    fun=neg_ll,
    x0=beta_init,
    method='BFGS',
    options={'disp': False}
)

beta_hat = result.x

# Hessian via finite differences for standard errors
from scipy.optimize import approx_fprime

n_params = len(beta_hat)
hessian = np.zeros((n_params, n_params))
eps = 1e-5
for i in range(n_params):
    def make_grad_i(idx):
        def grad_i(b):
            return approx_fprime(b, neg_ll, eps)[idx]
        return grad_i
    hessian[i, :] = approx_fprime(beta_hat, make_grad_i(i), eps)

cov_matrix = np.linalg.inv(hessian)
se = np.sqrt(np.diag(cov_matrix))

The customer coefficient is positive and statistically significant. After controlling for age and region, Blueprinty customers have a higher patent rate. In Poisson regression, coefficients work multiplicatively through the exponential link, so the rate ratio is \(e^{\hat{\beta}_{\text{customer}}}\), which tells you how many times higher the customer rate is relative to non-customers with the same characteristics.

The age terms show what I expected: a positive linear term and a negative quadratic, meaning patent rates rise with firm age and then taper off. That makes sense. Young firms are still building capability. Very old firms may have shifted priorities.

Every region coefficient is negative relative to the Northeast reference. So even after controlling for age and customer status, Northeast firms patent at higher rates. This matters. Remember, Blueprinty’s customers are disproportionately in the Northeast. Without the region controls, part of the “customer effect” would actually be a “Northeast effect.”

So what’s the actual estimated effect?

Coefficients from Poisson regression are multiplicative, not additive. Saying “the customer coefficient is 0.21” doesn’t directly tell you how many extra patents a firm gets. To translate this into something concrete, I use counterfactual predictions.

The logic: take all 1,500 firms with their actual ages and regions, predict their patent rate if none of them were customers, then predict again if all of them were. The average difference is the estimated effect.

Code 
# What if nobody was a customer?
X0 = np.array(X).copy()
X0[:, -1] = 0

# What if everybody was a customer?
X1 = np.array(X).copy()
X1[:, -1] = 1

y_hat_0 = np.exp(X0 @ beta_hat)
y_hat_1 = np.exp(X1 @ beta_hat)

ate = np.mean(y_hat_1 - y_hat_0)
Average predicted patents if nobody uses Blueprinty: 3.44
Average predicted patents if everybody uses Blueprinty: 4.23
Average marginal effect: 0.79 additional patents over 5 years

What to make of this

The model says that, holding age and region fixed, being a Blueprinty customer is associated with about 0.79 more patents over five years. That’s not nothing. For a firm averaging 3-4 patents, it’s a meaningful bump.

But I want to be careful about what this does and doesn’t tell us.

It does tell us that the customer-patent association survives after we account for age and geography. The raw gap between customers and non-customers isn’t purely an artifact of where they are or how old they are.

It does not tell us the software causes the difference. We controlled for two observable confounders. There are probably others we can’t see: R&D budget, number of engineers, management quality, how aggressive the firm is about IP. If firms that buy Blueprinty are also the ones that invest more in patenting generally, we’d see exactly this pattern even if the software did nothing.

Blueprinty’s marketing team could truthfully say their customers get more patents, even after adjusting for some firm characteristics. But “our customers patent more” is not the same as “our software makes you patent more.” The gap between those two statements is the gap between association and causation, and this dataset can’t close it.
